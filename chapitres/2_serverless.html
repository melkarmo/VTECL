Le Serverless Computing

 Le Serverless computing, ou informatique "sans serveur", est une méthode qui permet de fournir des services back-end en fonction de l'utilisation. Un fournisseur Serverless permet aux utilisateurs d'écrire et de déployer du code sans avoir à se soucier de l'infrastructure sous-jacente.

Petit rappel sur le back-end : Le développement d'applications est généralement divisé en deux domaines : le front-end et le back-end. Le front-end est la partie de l'application que les utilisateurs voient et avec laquelle ils interagissent, telle que la présentation visuelle. Le back-end est la partie que l'utilisateur ne voit pas ; cela comprend le serveur où les fichiers de l'application sont stockés et la base de données où les données de l'utilisateur et la logique commerciale sont conservées.

Schéma illustratif Front-End/Back-End [tonymarston]

Une entreprise qui obtient des services back-end d'un fournisseur Serverless est facturée sur la base de ses calculs et n'a pas à réserver et à payer une quantité fixe de bande passante ou de nombre de serveurs, car le service est automatiquement scalable. Il est à noter que bien qu'ils soient appelés "Serverless", les serveurs physiques sont toujours utilisés mais les développeurs n'ont pas besoin d'en être conscients.

Historique et contexte

Aux débuts du web, quiconque qui souhaitait construire une application web devait posséder le matériel physique nécessaire pour faire fonctionner un serveur, ce qui peut être très coûteux et lourd à gérer.

Puis est venu le Cloud computing, où des nombres fixes de serveurs ou des quantités d'espace serveur pouvaient être loués à distance. Les développeurs et les entreprises qui louent ces unités fixes d'espace serveur achètent généralement trop pour s'assurer qu'un pic de trafic ou d'activité ne dépassera pas leurs limites mensuelles et ne cassera pas leurs applications. Cela signifie qu'une grande partie de l'espace serveur payé est généralement gaspillé. Les fournisseurs de Cloud computing ont introduit des modèles de dimensionnement automatique pour résoudre ce problème, mais même avec le dimensionnement automatique, un pic d'activité non souhaité, comme une attaque DDoS, pouvait s'avérer très coûteux.

Technologie FaaS

En pratique, la plupart des fournisseurs Serverless (AWS, IBM, …) adoptent des technologies basées sur le FaaS.
Le Function as a Service (FaaS) est un moyen d'exécuter des morceaux de code modulaires sur un cloud. FaaS permet aux développeurs d'écrire et de mettre à jour un morceau de code à la volée, qui peut ensuite être exécuté en réponse à un événement, tel qu'un utilisateur qui clique sur un élément d'une application web. Il est ainsi facile d'adapter le code et constitue un moyen économique de mettre en œuvre des micro-services.

What is Serverless Computing ? - IDG.tv [NetworkWorld]

Pour rappel, les micro-services désignent à la fois une architecture et une approche de développement logiciel qui consiste à décomposer les applications en éléments les plus simples, indépendants les uns des autres. Diviser une application en micro-services est attrayant pour les développeurs car cela signifie qu'ils peuvent créer et modifier de petits morceaux de code qui peuvent être facilement mis en œuvre dans leurs bases de code. Cela s'oppose à l'architecture monolithique, dans laquelle tout le code est imbriqué dans un grand système. Avec les grands systèmes monolithiques, même une modification mineure de l'application nécessite un processus de déploiement lourd. FaaS élimine cette complexité de déploiement.

En utilisant un code Serverless comme FaaS, les développeurs web peuvent se concentrer sur l'écriture du code de l'application, tandis que le fournisseur Serverless s'occupe de l'allocation des serveurs et des services de maintenance.

Avantages

Coûts réduits - L'informatique Serverless est généralement très rentable, car les fournisseurs traditionnels de services de base (allocation de serveurs) font souvent payer à l'utilisateur l'espace inutilisé ou le temps d'inactivité du processeur.

Scalabilité simplifiée - Les développeurs qui utilisent une architecture Serverless n'ont pas à se soucier d'adapter le code à la montée de charge. Le fournisseur Serverless gère toute l'évolution à la demande.

Code back-end simplifié - Avec les FaaS, les développeurs peuvent créer des fonctions simples qui remplissent indépendamment un seul objectif, comme par exemple l'appel d'une API.

Délais d'exécution plus courts - L'architecture Serverless peut réduire considérablement le temps de mise sur le marché. Au lieu d'avoir besoin d'un processus de déploiement compliqué pour mettre en place des corrections de bogues et de nouvelles fonctionnalités, les développeurs peuvent ajouter et modifier le code au coup par coup.

Inconvénients

Moins de contrôle du système - La gestion d'une partie de l'infrastructure par une tierce partie rend difficile la compréhension de l'ensemble du système et ajoute des difficultés de débogage.

Plus de complexité requise pour les tests - Il peut être très difficile d'intégrer le code FaaS dans un environnement de test local, ce qui rend les tests approfondis d'une application plus difficiles.

Formation des développeurs - Pour pouvoir utiliser les infrastructures Serverless, il faut coder ses services back-end sous formes de FaaS, ce qui implique d’apprendre une nouvelle façon de développer pour s’adapter à la technologie.

Portabilité - La gestion de l’infrastructure étant assurée par une tierce partie, changer de provider implique souvent de modifier le code de ses fonctions puisque les technologies FaaS diffèrent d’un provider à un autre.

Sécurité - La gestion d'une partie de l'infrastructure par une tierce partie rend la sécurité de son application dépendante en grande partie de la sécurité garantie par le provider Serverless.
